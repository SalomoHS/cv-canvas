# CV Manager (cvvas)

Personal web application to manage and export CV/resume data. Separates content from presentation вЂ” edit once, export to PDF/DOCX with consistent styling.

## Project Overview

- **Purpose:** Centralized CV data management with multi-version support
- **7 fixed sections:** Profile в†’ About Me в†’ Education в†’ Professional Experience в†’ Organizational Experience в†’ Projects в†’ Skills (order is fixed, cannot be reordered)
- **Entry system:** Each entry has a status (`active` or `library`) вЂ” active entries appear in the CV, library entries are stored for later use
- **Multi-version:** Multiple CV versions can exist, each with different active entries and ordering
- **Export:** PDF (via `@react-pdf/renderer`) and DOCX (via `docx` npm package) with fixed templates

## Architecture

### Data Flow
- **Client в†’ API Route в†’ Prisma в†’ PostgreSQL**
- Zustand store (`src/store/useStore.ts`) provides React hooks and wraps all API calls
- All mutations go through API routes, never direct database access from client
- The store calls API routes, which use Prisma to read/write the database

### Key Directories
- `src/app/api/` вЂ” Next.js API routes (RESTful)
- `src/components/` вЂ” React components
- `src/store/` вЂ” Zustand store
- `src/lib/` вЂ” Utilities (Prisma client, types, export templates)
- `src/generated/prisma-client/` вЂ” Auto-generated Prisma client
- `prisma/` вЂ” Schema and migrations

### Database Models
- **Profile** вЂ” Name, contact, links (singleton)
- **Summary** вЂ” About Me paragraphs (multiple, selectable per version)
- **Entry** вЂ” Generic entry with `section` discriminator (`education`, `experience`, `project`, `skill`), `data` stored as JSON
- **Crate** вЂ” Groups related CVVersions together
- **CVVersion** вЂ” Links entries, stores ordering (`sectionOrder`, `skillOrder`)

### API Routes
All routes are under `src/app/api/`:
- `/api/profile` вЂ” GET/PUT
- `/api/entries` вЂ” GET/POST
- `/api/entries/[id]` вЂ” GET/PUT/PATCH/DELETE
- `/api/cv-versions` вЂ” GET/POST
- `/api/cv-versions/[id]` вЂ” GET/PUT/DELETE
- `/api/crates` вЂ” GET/POST
- `/api/crates/[id]` вЂ” GET/PUT/DELETE
- `/api/summaries` вЂ” GET/POST
- `/api/summaries/[id]` вЂ” GET/PUT/DELETE
- `/api/export/pdf` вЂ” POST (generates PDF blob)
- `/api/export/docx` вЂ” POST (generates DOCX blob)
- `/api/backup` вЂ” GET (export JSON), POST (import JSON)

### Docker Setup
- `docker-compose.yml` defines 3 services: `db` (postgres:16-alpine), `migrate` (runs migrations), `app` (Next.js standalone)
- Next.js is configured with `output: 'standalone'` for Docker
- Resource limits configured for all services

## Coding Conventions

- **Imports:** Use `@/` path alias (e.g., `@/lib/prisma`, `@/components/Sidebar`)
- **API routes:** Export named functions (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`)
- **Components:** Default exports, "use client" directive for interactive components
- **Types:** Defined in `src/lib/types.ts`, imported as type imports
- **Store:** Zustand with `create<StoreState>()`, API object for fetch wrappers
- **Styling:** Tailwind CSS utility classes; component files don't have separate CSS
- **Database IDs:** UUIDs generated by Prisma (`@default(uuid())`)
- **Entry data:** JSON column in database, typed via discriminated union in TypeScript

## Key Rules for Agents

1. **Next.js 16 specifics** вЂ” This project uses Next.js 16 (not 14/15). APIs and conventions may differ from older versions. Check actual code patterns before writing new code.
2. **Prisma client is at `@/generated/prisma-client`** (not `@prisma/client`) вЂ” The Prisma client is generated to `src/generated/prisma-client` using a custom output path.
3. **No localStorage** вЂ” Data is persisted in PostgreSQL, not localStorage. The original PRD mentioned localStorage-only, but the actual implementation uses a full database with API routes.
4. **Entry `data` is a JSON column** вЂ” When querying/updating entries, the `data` field is a Prisma JSON type. Read and write it as a plain JavaScript object.
5. **Section order is fixed** вЂ” The 7 sections always render in order: Profile, About Me, Education, Professional Experience, Organizational Experience, Projects, Skills. This cannot be changed.
6. **Experience has a discriminator** вЂ” `subType: "professional" | "organizational"` distinguishes the two experience types. Both share the same `ExperienceData` shape.
7. **Export styling is fixed** вЂ” PDF and DOCX templates have hardcoded styling (Times New Roman, 11pt, A4, etc.). No customization by users.
