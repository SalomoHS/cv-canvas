# CV Manager (cvvas)

Personal web application to manage and export CV/resume data. Separates content from presentation — edit once, export to PDF/DOCX with consistent styling.

## Project Overview

- **Purpose:** Centralized CV data management with multi-version support
- **7 fixed sections:** Profile → About Me → Education → Professional Experience → Organizational Experience → Projects → Skills (order is fixed, cannot be reordered)
- **Entry system:** Each entry has a status (`active` or `library`) — active entries appear in the CV, library entries are stored for later use
- **Multi-version:** Multiple CV versions can exist, each with different active entries and ordering
- **Export:** PDF (via `@react-pdf/renderer`) and DOCX (via `docx` npm package) with fixed templates

## Architecture

### Data Flow
- **Client → API Route → Prisma → PostgreSQL**
- Zustand store (`src/store/useStore.ts`) provides React hooks and wraps all API calls
- All mutations go through API routes, never direct database access from client
- The store calls API routes, which use Prisma to read/write the database

### Key Directories
- `src/app/api/` — Next.js API routes (RESTful)
- `src/components/` — React components
- `src/store/` — Zustand store
- `src/lib/` — Utilities (Prisma client, types, export templates)
- `src/generated/prisma-client/` — Auto-generated Prisma client
- `prisma/` — Schema and migrations

### Database Models
- **Profile** — Name, contact, links (singleton)
- **Summary** — About Me paragraphs (multiple, selectable per version)
- **Entry** — Generic entry with `section` discriminator (`education`, `experience`, `project`, `skill`), `data` stored as JSON
- **Crate** — Groups related CVVersions together
- **CVVersion** — Links entries, stores ordering (`sectionOrder`, `skillOrder`)

### API Routes
All routes are under `src/app/api/`:
- `/api/profile` — GET/PUT
- `/api/entries` — GET/POST
- `/api/entries/[id]` — GET/PUT/PATCH/DELETE
- `/api/cv-versions` — GET/POST
- `/api/cv-versions/[id]` — GET/PUT/DELETE
- `/api/crates` — GET/POST
- `/api/crates/[id]` — GET/PUT/DELETE
- `/api/summaries` — GET/POST
- `/api/summaries/[id]` — GET/PUT/DELETE
- `/api/export/pdf` — POST (generates PDF blob)
- `/api/export/docx` — POST (generates DOCX blob)
- `/api/backup` — GET (export JSON), POST (import JSON)

### Docker Setup
- `docker-compose.yml` defines 3 services: `db` (postgres:16-alpine), `migrate` (runs migrations), `app` (Next.js standalone)
- Next.js is configured with `output: 'standalone'` for Docker
- Resource limits configured for all services

## Coding Conventions

- **Imports:** Use `@/` path alias (e.g., `@/lib/prisma`, `@/components/Sidebar`)
- **API routes:** Export named functions (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`)
- **Components:** Default exports, "use client" directive for interactive components
- **Types:** Defined in `src/lib/types.ts`, imported as type imports
- **Store:** Zustand with `create<StoreState>()`, API object for fetch wrappers
- **Styling:** Tailwind CSS utility classes; component files don't have separate CSS
- **Database IDs:** UUIDs generated by Prisma (`@default(uuid())`)
- **Entry data:** JSON column in database, typed via discriminated union in TypeScript

## Key Rules for Agents

1. **Next.js 16 specifics** — This project uses Next.js 16 (not 14/15). APIs and conventions may differ from older versions. Check actual code patterns before writing new code.
2. **Prisma client is at `@/generated/prisma-client`** (not `@prisma/client`) — The Prisma client is generated to `src/generated/prisma-client` using a custom output path.
3. **No localStorage** — Data is persisted in PostgreSQL, not localStorage. The original PRD mentioned localStorage-only, but the actual implementation uses a full database with API routes.
4. **Entry `data` is a JSON column** — When querying/updating entries, the `data` field is a Prisma JSON type. Read and write it as a plain JavaScript object.
5. **Section order is fixed** — The 7 sections always render in order: Profile, About Me, Education, Professional Experience, Organizational Experience, Projects, Skills. This cannot be changed.
6. **Experience has a discriminator** — `subType: "professional" | "organizational"` distinguishes the two experience types. Both share the same `ExperienceData` shape.
7. **Export styling is fixed** — PDF and DOCX templates have hardcoded styling (Times New Roman, 11pt, A4, etc.). No customization by users.
